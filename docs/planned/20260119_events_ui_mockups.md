## Design Decisions

### 1. No Event Points

| Decision | Reasoning |
|----------|-----------|
| Events do NOT have points | Events are grouping containers, not tasks |
| Chores inside events have points | Existing point system unchanged |
| Future option | Optional "completion bonus" if users request |

**Why no event points:**
- Avoids double-counting confusion
- Simpler mental model for kids
- Fewer edge cases (partial completion, chore removal)
- Events = organization, not gamification

### 2. No Health Badges

| Decision | Reasoning |
|----------|-----------|
| Skip health badges for MVP | Redundant when chores are visible |
| Chore checkboxes show status | âœ…/â˜ already indicates completion |
| Add later if requested | Validate demand before building |

**Why no health badges:**
- When chores are expanded, you can already see completion status
- Extra API endpoint for marginal value
- 10% effort for 5% value - not Pareto
- Core value is **grouping**, not badges

### 3. Dedicated /parent/events Page

| Decision | Reasoning |
|----------|-----------|
| Events get own page | Settings = config, Events = content |
| Matches /reports pattern | Consistent navigation mental model |
| Not in settings tab | Events are managed content, not configuration |

### 4. Kid-Facing Language: "Missions"

| Audience | Word | Reasoning |
|----------|------|-----------|
| Kids | "Missions" | Adventure framing, not drudgery |
| Parents | "Chores" | Familiar, practical |
| Database/API | `chores` | No refactor needed |

**Research-backed**: Identity-based language ("be a helper") works better than action language ("help") with children.
