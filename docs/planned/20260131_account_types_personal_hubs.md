## Executive Summary

This document outlines a strategic expansion of ChoreGami to serve three distinct user types beyond the current "family with kids" model:

1. **Family with Kids** (current) - Gamified chores with parent oversight
2. **Roommates / Couples** - Peer accountability without gamification
3. **Just Me** (Solo) - Personal task management for teens, students, adults

The architecture leverages **existing user switching** (right nav) as the personal view mechanism, with the **home/index page as the shared hub**. No new "My View vs Hub" toggle is needed.

---

## Problem Statement

### Current Limitation
ChoreGami is optimized exclusively for families with young children:
- Points and leaderboards assume kids competing
- Parent/child roles don't fit roommates or couples
- Single users (teens, students) have no solo mode
- Gamification can feel infantilizing to adults

### Market Opportunity
Competitors like Cozi serve families but poorly handle:
- **Roommates**: Need fairness tracking, not gamification
- **Teens**: Want homework/schedule tracking, not "kid" UX
- **College students**: Need simple task lists, not family features
- **Couples**: Want shared lists without parental controls

## Data Model: One Schema, Multiple Modes

Uses existing JSONB settings architecture.
**See**: [JSONB Settings Architecture](../20260114_JSONB_settings_architecture.md)

### families Table Extension

```jsonc
// families.settings
{
  // NEW: Account type determines defaults
  "account_type": "family" | "roommates" | "personal",

  // Existing fields continue to work
  "theme": "fresh_meadow",

  "apps": {
    "choregami": {
      // Existing settings
      "children_pins_enabled": false,

      // NEW: Mode-specific settings
      "gamification_enabled": true,      // false for roommates
      "fairness_tracking": false,        // true for roommates
      "bill_splitting": false,           // true for roommates
      "hub_name": "Family Hub"           // or "Apartment", null for solo
    }
  }
}
```

### family_profiles Table Extension

```jsonc
// family_profiles.preferences
{
  // Existing fields
  "theme": "ocean_depth",

  // NEW: Role simplification
  // For roommates: all members have role = "member" (equal peers)
  // For personal: single profile, no role needed

  // NEW: Age-based gamification
  "age_group": "kid" | "teen" | "adult",

  // NEW: Per-user gamification opt-out
  "gamification": {
    "show_points": true,     // Teen might opt out
    "show_streaks": true,    // Personal motivation
    "show_leaderboard": true // Might hide for privacy
  }
}
```

### Router Logic

```typescript
// In /login.tsx or /register.tsx POST handler

// Check for invite token first (highest priority)
const inviteToken = formData.get("invite_token") ||
                    localStorage.get("pendingInviteToken");
if (inviteToken) {
  const invite = await InviteService.findByToken(inviteToken);
  if (invite) {
    return redirect(`/setup/join?token=${inviteToken}`);
  }
}

// Check for referral code (pre-select but editable)
const refCode = url.searchParams.get("ref");
if (refCode) {
  const referrer = await ReferralService.findByCode(refCode);
  if (referrer) {
    return redirect(`/setup/referred?ref=${refCode}`);
  }
}

// No context - full persona selection
return redirect("/setup/new");
```

### Edge Cases

| Scenario | Handling |
|----------|----------|
| Referral + Invite both present | Invite wins (joining > creating) |
| Expired invite token | Treat as organic â†’ `/setup/new` |
| Invalid referral code | Silent fallback â†’ `/setup/new` |
| Referrer changed account type | Use type at code creation (immutable) |
| "Personal" referral | Pre-select "Just Me", allow change |

### Benefits

| User Type | Benefit |
|-----------|---------|
| **Invited users** | Zero friction - just enter name and join |
| **Referred users** | Smart default, still flexible |
| **Organic users** | Clear choice upfront |

**Principle**: Decisions only when necessary. Most users come via invite or referral (social spread), so optimize for those paths.

### Required Changes: `invite-service.ts`

**Current Role Type**:
```typescript
role: "parent" | "child"  // parent = co-parent, child = teen
```

**Proposed Role Type**:
```typescript
role: "parent" | "child" | "member"  // member = equal peer (roommates)
```

**Email Template Changes**:

### Required Changes: `/join.tsx`

**Current Behavior**:
- Creates profile with `role` from invite
- Only supports "parent" or "child"

**Proposed Changes**:
- Support "member" role for roommate invites
- For roommates: all invitees get `role: "member"`
- Show different welcome messaging based on account_type

### Referral Service: No Changes Needed

The referral system works at the `family_id` level. All account types create a family record (even solo users), so:
- `/r/[code]` â†’ `/register?ref=CODE` works unchanged
- `ReferralService.recordConversion()` tracks by family_id
- Attribution works identically for all account types

### OAuth Flow: No Changes Needed

OAuth flow is account-type agnostic:
1. User clicks OAuth provider
2. Supabase creates auth user
3. Redirect to `/setup`
4. `/setup` shows account type selector (NEW)
5. User completes form based on type

The only change is in `/setup.tsx` - OAuth callbacks remain unchanged.

### Backward Compatibility

**Existing families**:
- No `account_type` in settings â†’ default to "family"
- Gamification remains enabled
- No migration needed

**Query Pattern**:
```typescript
const accountType = family.settings?.account_type || "family";
const gamificationEnabled = accountType === "family";
const fairnessTracking = accountType === "roommates";
```

## Implementation Phases

### Phase 1: Persona-Typed Metadata (Low Effort)
1. Add `account_type` field to `PendingInvite` interface
2. Add `account_type` field to referral code metadata
3. Auto-populate from `family.settings.account_type` on creation
4. Add "member" to role enum in invite-service.ts

### Phase 2: Route Splitting (Medium Effort)
1. Create `/setup/join` - minimal form, inherited context
2. Create `/setup/referred` - pre-selected, editable
3. Create `/setup/new` - full persona selection (current `/setup` becomes this)
4. Add router logic in login/register POST handlers

### Phase 3: Route-Specific UX (Medium Effort)
1. `/setup/join`: Name-only form, contextual messaging
2. `/setup/referred`: Pre-selected toggle, dynamic form fields
3. `/setup/new`: Full persona cards with descriptions
4. Dynamic email templates based on account_type

### Phase 4: Dashboard Adaptations (Medium Effort)
1. Fairness tracking for roommates (replace leaderboard)
2. Simplified nav for solo users (no hub toggle)
3. Age-based gamification toggles
4. Empty state variations per account type

### Phase 5: Polish (Low Effort)
1. Welcome messaging per route
2. Onboarding hints ("Add roommates" vs "Add kids")
3. Settings page "Convert account type" option
4. Help documentation per persona
